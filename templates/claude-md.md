## 测试用例覆盖要求

### 基础操作覆盖

- 每个功能模块必须覆盖页面进入、数据加载、基本交互等基础操作用例。
- 包含菜单导航、页面元素展示、基础查询等场景。

### 核心链路覆盖

- 必须覆盖业务端到端完整流程，从触发条件到最终结果的完整链路。

### 边界值覆盖

- 测试数据必须包含边界值场景：
  - **最大值**：如数据量极大（10,000+条）、字段值达到上限。
  - **最小值**：如数据量极少（仅1条）、字段值为0或最小有效值。
  - **空值**：如无数据时页面展示、必填字段为空、关联数据缺失。

### 异常场景覆盖

- 必须覆盖异常与错误场景：
  - **数据缺失**：源系统数据不完整、关联表记录缺失。
  - **服务不可用**：外部系统接口故障、网络断开。
  - **操作异常**：重复操作、并发操作、超时操作。
  - **配置缺失**：邮箱未配置、定时任务未部署、角色权限未分配。

### 覆盖率要求

- 最终产出的测试用例覆盖率须达到 **80% 及以上**。
- 覆盖率计算：已覆盖的需求功能点数 / 需求文档中总功能点数 × 100%。

## 测试用例输出模板

### 模板文件

- 路径：`测试用例/测试用例模板.xlsx`
- 全量模式输出：`测试用例/sprint_all/testCase.xlsx` + `测试用例/sprint0/sprint0_testCase.xlsx`
- 增量模式输出：`测试用例/sprintN/sprintN_testCase.xlsx`

### 列定义与顺序

必须严格遵循模板的列定义与顺序，共 10 列：

| 测试用例ID | 模块 | 标题 | 前置条件 | 测试数据 | 测试步骤 | 预期结果 | 实际结果 | 覆盖功能点 | 关联接口 |
| ---------- | ---- | ---- | -------- | -------- | -------- | -------- | -------- | ---------- | -------- |

- 实际结果列默认为空。

### 用例ID生成规则

- 格式：`{前缀}_TC_{模块缩写}_{序号}`
- 序号从 `001` 开始。

### 模块字段规则

- **来源**：从需求文档中识别功能模块划分。
- **命名**：使用需求文档中的模块名称，保持原样，若文档未明确划分则由分析模型根据功能聚类自动命名。
- **与 Sheet 的关系**：一个模块对应一个 Sheet，Sheet 名称与模块字段值保持一致。
- **用例归属**：每条测试用例的"模块"列填写其所属模块名称，同一模块下所有用例该列值相同。

#### Sheet 拆分规则

- 按 PRD 中的功能模块拆分为多个 Sheet。
- 若只有一个模块，则使用单一 Sheet。

### 标题字段规则

- **来源**：从需求文档的具体功能点、业务场景或用户操作路径中提炼。
- **格式**：简明扼要，体现测试意图。
- **编写要求**：
  - 正向场景：描述正确操作后的预期结果。
  - 异常场景：描述触发条件及系统响应。
  - 边界场景：描述边界值或临界条件。
- **唯一性**：同一模块内标题应保持唯一，避免重复。

### 前置条件字段规则

- **定义**：执行该测试用例前必须满足的环境、数据或状态条件。
- **编写要求**：
  - 使用简洁的陈述句，多条条件用序号或分号分隔。
  - 包含但不限于：用户登录状态及角色权限、系统配置、测试数据准备、依赖的其他功能或任务状态。

### 测试数据字段规则

- **定义**：执行测试用例时需要输入或使用的具体数据内容。
- **编写要求**：
  - 明确列出数据内容，多条数据用分号或换行分隔。
  - 若无需特定测试数据，可填写"无"或留空。
  - 若涉及具体数值、日期、编码，尽量给出具体示例。

### 测试步骤字段规则

- **定义**：描述执行测试用例时需要依次进行的操作动作，从用户登录界面开始。
- **格式要求**：
  - 使用有序列表（数字序号），每一步为一个独立操作。
  - 步骤必须清晰、可执行，避免模糊描述。
- **起点要求**：
  - 第一条操作必须为：`1. 用户输入用户名、密码点击登录。`
- **操作数量**：
  - 每个操作项编号递增，步骤数量应与预期结果项数量保持一致。

### 预期结果字段规则

- **定义**：描述测试步骤中每一项操作对应的预期系统响应或状态变化。
- **对应原则**：测试步骤与预期结果一一对应，项数相同。
- **格式要求**：使用有序列表，编号与测试步骤严格对应。
- **内容要求**：每项结果应明确可验证。

### 样式要求

- 保留模板文件的样式（表头加粗、自动列宽等）。

## E2E 自动化测试脚本

### 目标系统

- 地址：`{系统地址}`
- 测试账号：`{账号}` / `{密码}`

### 脚本存放路径

- 全量模式脚本：`测试用例/sprint_all/scripts/`
- 增量模式脚本：`测试用例/sprintN/sprintN_scripts/`
- 选择器映射：`{脚本路径}/selectors/{模块名}.selectors.ts`
- 测试脚本：`{脚本路径}/{模块名}.spec.ts`
- 文件名与模块名一致，方便按模块排查。

### 生成策略（混合方案）

采用"录制补充选择器 + 文本批量生成"的混合方案：

1. **录制核心链路**：对每个模块，用 Playwright MCP 录制 2-3 条核心用例，提取真实选择器，生成 `.selectors.ts` 映射文件。
2. **批量生成脚本**：基于 Excel 测试用例描述 + 已验证的选择器映射，批量生成全部用例的 `.spec.ts` 脚本。
3. **执行修复**：运行脚本，对失败用例进行修复（最多 3 轮），仍失败的标记 `test.skip()` 并注释原因。

### 选择器优先级

以 `browser_snapshot` 与 `templates/extract-selectors.js` 的实际结果为准；可优先使用语义 locator，但必须通过真实页面结构验证。

## V2 关联图谱约定

> 由 `bf-graph-agent` 子 agent 调用 `~/.claude/skills/scripts/build_index.py` 维护。
> 本节落档的是「规范层」约定：所有功能点 / 用例 / 接口 / 脚本块 / 业务规则 / 端到端流程必须显式带全局 ID，作为图谱解析的地基。

### 1. 功能点锚点（FP）

每个 `功能点.md` 的功能点标题**必须**带 HTML 注释锚点，序号在模块内连续、两位补零：

```markdown
## 功能点1：选择商品 <!--FP_XD_01-->

- **描述**：……
```

- 锚点格式：`<!--FP_{模块缩写}_{两位序号}-->`（如 `FP_XD_01`、`FP_XD_02`、`FP_CGTJ_10`）
- 模块缩写必须与 `cases.json` 用例 ID 中的缩写一致（如 `SPD_TC_XD_001` 对应 `FP_XD_01`）
- 锚点是纯 HTML 注释，对 V1 脚本与人类阅读完全无害
- 解析正则（供 `build_index.py` 使用）：`## 功能点(\d+)：.*?<!--(FP_[A-Z]+_\d+)-->`

### 2. cases.json 可选字段：covers / tests_api

每条用例可选附加两个字段，缺省视为 V1 兼容（不报错）：

| 字段 | 类型 | 含义 | 缺省 |
|------|------|------|------|
| `covers` | `string[]` | 该用例覆盖的功能点 FP 锚点列表（如 `["FP_XD_01"]`） | `[]` |
| `tests_api` | `string[]` | 该用例验证的接口节点 ID 列表（无则 `[]`） | `[]` |

示例：

```json
[
  {
    "id": "SPD_TC_XD_001",
    "module": "下单",
    "title": "选择商品后进入结算页",
    "precondition": "1. 已登录",
    "test_data": "商品SKU=1001",
    "steps": "1. 用户输入用户名、密码点击登录。\n2. 进入下单页选择商品。",
    "expected": "1. 登录成功。\n2. 跳转结算页并展示商品。",
    "covers": ["FP_XD_01"],
    "tests_api": []
  }
]
```

### 3. 节点 ID 规则表

| 节点类型 | 前缀 | ID 格式 | 来源 |
|---------|------|--------|------|
| 功能点 | `fp` | `FP_{模块缩写}_{两位序号}` | 功能点.md 标题锚点 |
| 测试用例 | `case` | `{项目前缀}_TC_{模块缩写}_{三位序号}` | cases.json `id` |
| 接口 | `api` | `API_{大写下划线名}`（如 `API_ORDER_CREATE`） | 接口功能点.md（P3 引入） |
| 脚本块 | `script` | `SCRIPT_{模块}_{用例ID}` | spec.ts `test('ID - title')` |
| 业务规则 | `rule` | `RULE_{模块缩写}_{两位序号}` | 功能点.md `业务规则` 提取 |
| 端到端流程 | `flow` | `FLOW_{流程名}` | PRD 流程章节（P2 引入） |
| 页面 | `page` | `PAGE_{大写下划线名}` | bf-ui-explorer 报告（P3 引入） |

### 4. 图谱产物固定位置（唯一真相源）

每项目一个本地 SQLite 库，与其他项目天然隔离：

```
{项目根}/需求文档/sprint_all/索引/
├── 知识图谱.db              # SQLite 数据库（nodes / edges / flows / unresolved_deps）
├── 知识图谱.json            # 节点/边 dump，便于人工查看
├── 覆盖率报告.md            # 各模块功能点覆盖情况（P1 产出）
├── 流程依赖图.md            # Mermaid 端到端流程图（P2 产出）
├── 待确认依赖.md            # 启发式推断但需人工确认的依赖（P2 产出）
└── 影响面报告_sprintN.md    # 增量回归时按 sprint 产出（P2 产出）
```

**铁律**：图谱只在 `需求文档/sprint_all/索引/` 创建与查询。**禁止**在 `sprint0/`、`sprintN/`、项目根或任何子模块目录创建/查询图谱——避免数据冗余与多份不一致。`build_index.py` 的 `--source` 参数锁死 `sprint_all` 单值。
